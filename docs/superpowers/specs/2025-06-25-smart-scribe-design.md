# Smart Scribe — 设计规格

> 带证据链的课堂/会议记录工作台（自托管 Web 工具）
> 日期: 2025-06-25 | 状态: 定稿待实现

---

## 1. 产品定位

上传视频/音频/图片任意组合 → 自动语音转写、PPT 画面 OCR、AI 结构化总结，且**原文和总结永远共存**。每个 AI 结论都可追溯到具体的 PPT 页或语音时间点。

核心设计原则：

- **原文优先** — 转写和 OCR 原文完整保留，AI 总结只是辅助
- **可追溯** — 每个结论都能回到具体 PPT 页/语音时间点
- **不盲信 AI** — 列出未被引用的片段，人机协作
- **过程可见** — AI 处理过程不是黑盒，用户能看到每一步在做什么

## 3. 处理流水线

```
素材输入（本地上传 / 链接下载 → yt-dlp/Playwright 拉取）
        │
        ▼
┌──────────────────┐
│ 1. 素材分类入库   │  视频/音频/图片自动识别，写入 SQLite
│    保留上传顺序    │  sort_order 字段记录用户拖拽/粘贴顺序
└────────┬─────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌──────────┐ ┌──────────┐
│2a. 视频   │ │2b. 图片   │  并行处理
│ffmpeg    │ │直接 OCR   │
│每隔N秒抽帧│ │PaddleOCR  │
│→ 去重    │ │PP-OCRv6   │
│→ OCR     │ │           │
└────┬─────┘ └────┬─────┘
     │            │
     │       ┌────┘
     ▼       ▼
┌──────────────────┐
│ 3. 音频提取       │  视频 → ffmpeg 抽音轨（16kHz 单声道 WAV）
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 4. 语音转写       │  火山引擎豆包语音妙记
│    说话人分离      │  返回: 时间戳 + 说话人 + 文本片段
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 5. 证据块生成     │  S001/S002...(语音) + P001/P002...(画面)
│    + 自动匹配     │  匹配算法: 时间邻近(0.35) + Jaccard(0.3) + 语义余弦(0.35)
│                   │  时间窗口: 前后各 2 分钟
│                   │  阈值: 加权分 >0.6 自动关联
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 6. AI 总结        │  DeepSeek 输入: 原始转写 + OCR + 证据块 + 场景上下文
│    + 纠错         │  输出 JSON: { corrected_text, summary, key_points, citations[] }
│    + 引用校验      │  后端验证每个 citation.block_id 真实存在于数据库
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 7. 未被引用片段    │  扫描所有证据块，未被任何 citation 引用的列入遗漏列表
└──────────────────┘
```

### 3.1 PPT 最优页选择策略

- 视频 → ffmpeg 每隔 N 秒（默认 3 秒）抽一帧
- 去重：相邻帧像素相似度 >95% 的视为同一画面，只保留第一帧
- 去重后的帧逐一 OCR
- 选择 OCR 文本字符数最多的帧作为 P001 主证据页
- 其他去重帧如有显著不同内容（文字量 > 主帧的 50%），也保留为补充页 P002/P003...

### 3.2 AI 纠错策略

- 转写原文**不修改**，完整保留在数据库
- 纠错阶段：DeepSeek 收到原文 + OCR + 场景上下文后，判断同音字/领域术语错误并输出修正版
- 纠错结果标注修改位置（offset + old_text + new_text），前端可高亮显示变更
- 纠错、摘要、要点在**同一次 API 调用**中完成，减少请求次数

### 3.3 AI 输出结构（单次 DeepSeek 调用）

```json
{
  "corrected_text": "纠错后的完整转写文本，标注修改位置",
  "summary": "结构化摘要，按主题分段落，每段带 citations",
  "key_points": [
    { "point": "核心要点1", "citations": ["S001", "P003"] },
    { "point": "核心要点2", "citations": ["S005"] }
  ],
  "corrections": [
    { "offset": 42, "old": "电词", "new": "电磁" }
  ]
}
```

## 4. 数据模型

```
Session（会话）
├─ id, title, status, created_at, updated_at
│
├─ 一对多 → Material（素材）
│            ├─ type: video/audio/image
│            ├─ source: local_file/url
│            ├─ file_path, original_url, sort_order
│            ├─ status: pending/processing/done/error
│            └─ 一对多 → EvidenceBlock（证据块）
│                        ├─ block_id: 全局唯一 "S001"/"P001"
│                        ├─ type: speech/screen
│                        ├─ timestamp, end_timestamp
│                        ├─ speaker (仅 speech)
│                        ├─ text
│                        ├─ page_number (仅 screen)
│                        ├─ image_path (仅 screen，缩略图)
│                        └─ 多对多 → Match（匹配关系）
│                                    ├─ score
│                                    ├─ time_sim / keyword_sim / semantic_sim
│
├─ 一对一 → Transcript（转写原文，不作修改）
│            └─ 一对多 → TranscriptSegment
│                        ├─ start_time, end_time
│                        ├─ speaker, text
│
└─ 一对一 → Summary（AI 输出）
             ├─ corrected_text (JSON，含 modifications)
             ├─ summary_markdown
             ├─ key_points (JSON)
             ├─ citations (JSON，引用列表)
             ├─ unused_block_ids (JSON，未被引用块)
```

### 关键字段说明

- `sort_order`: 保留用户上传/粘贴的原始顺序，用于时间线排列和匹配
- `block_id`: 全局唯一，S 系列和 P 系列分别自增编号
- `Match`: 存储自动匹配结果，支持用户手动调整关联关系
- `Summary`: AI 返回内容以原生 JSON 存储，拆字段方便前端查询和渲染

## 7. 证据链系统

### 7.1 证据块定义

- **S 系列（Speech）**：来自语音转写，包含时间戳、说话人、文本
- **P 系列（Picture/Screen）**：来自画面 OCR，包含页码、时间戳、缩略图、OCR 文本

编号全局唯一，自增。如 S001、S002、P001、P002。

### 7.2 自动匹配算法

输入：所有 S 块和 P 块
对每一对 (S, P)：

```
total_score = 0.35 × time_sim + 0.30 × jaccard_sim + 0.35 × semantic_sim

time_sim = max(0, 1 - |S.time - P.time| / 120)   // 2分钟窗口
jaccard_sim = |S.keywords ∩ P.keywords| / |S.keywords ∪ P.keywords|
semantic_sim = cosine(text_embedding(S), text_embedding(P))
                                                  // 使用 DeepSeek embedding
```

total_score > 0.6 → 自动关联为匹配对
用户可在前端手动调整匹配关系

### 7.3 引用校验

AI 总结返回的每个 citation 必须：
- block_id 在数据库中存在
- block_id 属于当前 session
- 校验失败 → 前端标记该引用为无效（红色），不计入可用引用

### 7.4 未被引用片段

所有证据块 - 被任意 citation 引用的块 = 未被引用片段列表
在右侧面板底部灰度显示，供用户巡检是否有遗漏重要内容。

## 8. 下载模块

### 8.1 设计原则

- **轻量内置**，不引入独立 Agent 服务
- yt-dlp 处理主流视频平台（抖音、B站、YouTube）
- Playwright 最小封装处理特殊格式（抖音图文等）
- 下载结果统一归一化为本地文件（视频 mp4 / 图片 jpg,png / 音频 mp3），进入标准流水线

### 8.2 链接处理流程

```
用户粘贴链接 → 识别平台 → 选择下载策略
                           ├─ 视频平台 → yt-dlp → 本地 mp4
                           ├─ 图文帖子 → Playwright 截图/提取 → 本地图片
                           └─ 直接文件链接 → HTTP 下载
→ 写入 Material 表 → 进入标准流水线
```

### 8.3 yt-dlp 支持清单

| 平台 | 格式 | 工具 |
|------|------|------|
| 抖音 | 视频 | yt-dlp |
| B站 | 视频 | yt-dlp |
| YouTube | 视频 | yt-dlp |
| 抖音图文 | 图片+文字 | Playwright |
| 直接链接 | .mp4/.mp3/.jpg 等 | HTTP 直接下载 |

## 9. 设置面板（API 配置）

用户首次使用需配置外部 API key。设置面板独立于工作区，通过顶栏齿轮图标进入。

### 9.1 配置项清单

| 配置项 | 是否必填 | 说明 |
|--------|----------|------|
| 火山引擎语音 API | **必填** | app_id + access_token（豆包语音妙记） |
| DeepSeek API | **必填** | api_key（OpenAI 兼容格式） |
| PaddleOCR | 默认本地，可选切云端 | 本地零配置；切云端 VL 需填 AI Studio key |
| yt-dlp cookie | 可选 | B站/抖音登录态下载所需 |

### 9.2 存储与安全

- API key 存储在 SQLite 单独表中，加密存储（AES-256，密钥基于部署时生成的随机 secret）
- 密钥**不出现在前端代码中**，后端代理所有外部 API 调用
- 前端设置面板提交时走 HTTPS，自托管环境默认 HTTP 亦支持
- 提供"测试连接"按钮，逐一验证 key 有效性

### 9.3 OCR 部署模式切换

```
设置面板 → OCR 模式选择
  ├─ 本地 PaddleOCR PP-OCRv6（默认）
  │   └─ 后端 from paddleocr import PaddleOCR 直接调用
  │   └─ 零 key 零费用，首次加载较慢，CPU 推理
  └─ 云端 PaddleOCR-VL
      └─ 后端走 AI Studio API，需要 API key
      └─ 每天免费 2 万页，效果优于本地，速度快
```

切换后立即生效，下一次 OCR 任务按新模式执行。

### 9.4 商业化预留（V1 不实现）

V1 采用用户自带 key 模式，零成本验证产品。未来如转付费：
- 后端预留 key 管理模块，可注入平台 key 替代用户 key
- 前端预留"使用平台 API"开关位
- 不影响现有流水线代码，仅在 API 调用层切换 key 来源

---

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 来源处理 | 轻量内置下载模块 | 保持项目独立性，不引入重 Agent |
| PPT 策略 | 抽帧去重→OCR→选文字最多 | 平衡准确度和性能 |
| 语音纠错 | 总结时顺带纠错，原文不动 | 可追溯，原味保留 |
| AI 输出 | 三层：纠错原文+摘要+要点 | 覆盖完整使用场景 |
| 纠错+总结 | 同一次 DeepSeek 调用 | 减少 API 成本，语义连贯 |
| API 架构 | 三大模块：媒体/语音/AI | 职责清晰，可独立开发 |
| 进度通信 | WebSocket | 实时推送，前端无轮询 |
| 前端布局 | 三栏 Pr 风格（220/55%/30%） | 专业感，信息密度合理 |
| 配色 | 深色工具体系 | Adobe Pr/VSCode 风格 |
| 数据库 | SQLite | 自托管友好，无需额外服务 |
